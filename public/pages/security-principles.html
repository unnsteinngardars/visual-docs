Security Principles

Foundational concepts that guide every security decision

The CIA Triad

The CIA triad is the foundational model for information security. Every control, policy, and design decision maps back to one (or more) of its three pillars: Confidentiality, Integrity, and Availability. A strong security posture keeps all three in balance.

C
Confidentiality
Only the right eyes
Ensures information is accessible only to those explicitly authorised to see it. Data stays private whether it's at rest, in transit, or in use.
  • Encryption — TLS, AES, full-disk crypto
  • Access control — RBAC, least privilege
  • Authentication — MFA, strong passwords
  • Data classification — handle by sensitivity
  • Broken by: data breaches, eavesdropping, shoulder surfing
I
Integrity
Unaltered & trustworthy
Guarantees data and systems remain accurate and unmodified except by authorised parties. If a byte changes, you'll know — and you'll know who.
  • Hashing — SHA-256, HMAC verification
  • Digital signatures — non-repudiation
  • Checksums — detect corruption in transit
  • Version control & audit logs
  • Broken by: tampering, MITM injection, insider modification
A
Availability
There when you need it
Ensures authorised users can reach systems and data whenever they need them. Security that blocks legitimate access is security that has failed.
  • Redundancy — HA clusters, multi-region
  • Backups & DR plans
  • DDoS protection — rate limits, WAF, CDN
  • Patching & capacity planning
  • Broken by: ransomware, DoS attacks, hardware failure
Trade-offs The three pillars often pull against each other. Encrypting every backup (confidentiality) can slow recovery (availability). Strict change control (integrity) can delay incident response. Good security isn't maximising one pillar — it's choosing the right balance for the threat model and business need.

The DAD Triad

If CIA describes what defenders protect, the DAD triad describes what attackers — or disasters — inflict. Every offensive outcome maps to one of Disclosure, Alteration, or Destruction / Denial. It's the mirror image of the CIA triad, and a useful lens for threat modelling.

D
Disclosure
Breaks Confidentiality
Sensitive information reaches unauthorised parties. The data itself may be unchanged — the damage is that it was seen at all.
  • Data breach — DB dump, S3 bucket leak
  • Eavesdropping — unencrypted Wi-Fi, MITM
  • Credential theft — phishing, keyloggers
  • Insider leak — screenshots, emails
  • Driver: espionage, extortion, competitive advantage
A
Alteration
Breaks Integrity
Data or systems are modified without authorisation. The values you trust are no longer the values that were written — and you may not notice.
  • Tampering — forged transactions, records
  • Defacement — modified web content
  • Code / supply-chain injection
  • Log manipulation — covering tracks
  • Driver: fraud, sabotage, anti-forensics
D
Destruction / Denial
Breaks Availability
Data is wiped or systems are rendered unreachable. The asset still belongs to you — you just can't use it.
  • Ransomware — encrypts to deny access
  • Wiper malware — irreversible destruction
  • DDoS — overwhelms the service
  • Hardware failure / outage
  • Driver: disruption, extortion, hacktivism
Confidentiality ⇄ Disclosure
Integrity ⇄ Alteration
Availability ⇄ Destruction

Security Models

Formal models that describe how systems should enforce confidentiality, integrity, or access control. Each one picks a specific property to protect and defines the rules that keep it intact — useful vocabulary for designing or evaluating any security architecture.

Bell–LaPadula
Confidentiality
The classic military-grade multi-level security model. Subjects and objects get classification labels; information is never allowed to flow from a higher level to a lower one.
No read up (simple security) · no write down (★-property)
Biba
Integrity
The integrity counterpart of Bell–LaPadula. Prevents lower- integrity data from corrupting higher-integrity data — an analyst can't let untrusted input overwrite a trusted report.
No read down (simple integrity) · no write up (★-integrity)
Clark–Wilson
Integrity
Commercial-world integrity model built around well-formed transactions: users can only modify data through certified programs, and separation of duties prevents a single actor from committing fraud.
Subject → TP → CDI · enforces certification & separation of duties
Brewer & Nash
Conflict of Interest
Also called the Chinese Wall model. Access rights change dynamically based on what a subject has already accessed — once you view client A's data, data from competing client B becomes off-limits.
History-based access · prevents insider conflicts of interest
Goguen–Meseguer
Non-interference
Earliest formal non-interference model. Actions of high-level users must not influence what low-level users can see — pre- defined permission lists decide exactly which subjects may interact with which objects.
High actions invisible to low subjects
Sutherland
Information Flow
State-machine model focused on preventing unauthorised inference — stopping an attacker from deducing secret state by observing permitted outputs.
Blocks covert channels & inference attacks
Graham–Denning
Access Control
Defines the eight primitive commands needed to securely create and delete subjects & objects, and to assign or transfer rights between them — foundation for most OS-level access control systems.
8 primitives: create/delete subject, create/delete object, read/grant/delete/transfer right
Harrison–Ruzzo–Ullman (HRU)
Access Control
Extends Graham–Denning with an access-control matrix and formal commands. Famously proved the safety problem — determining whether a right can ever leak — is undecidable in the general case.
Access matrix · safety is undecidable

ISO/IEC 19249

ISO/IEC 19249 catalogues principles for building secure systems. It splits them into architectural principles (how the system is structured) and design principles (how components behave) — five of each.

Architectural Principles 5 · Structure

How the system is put together.

Domain Separation
Group related components into isolated domains, each with its own security attributes. Example: x86 privilege rings — kernel in ring 0, user apps in ring 3. Ties into the Goguen–Meseguer model.
Layering
Structure the system into abstract layers so policies can be enforced and validated at each level. Example: the seven-layer OSI model. This is the basis of Defence in Depth.
Encapsulation
Hide internals and expose only controlled methods or APIs. Example (OOP): call clock.increment() instead of writing directly to the seconds field — prevents invalid state.
Redundancy
Duplicate components so failure of one doesn't take the system down, and tampering can be detected. Examples: dual power supplies, RAID 5 parity. Protects both availability and integrity.
Virtualization
Share hardware across isolated guests. Provides sandboxing, safe malware detonation, and stronger security boundaries — the foundation of modern cloud and analysis labs.

Design Principles 5 · Behaviour

How components behave at runtime.

Least Privilege
Give each subject the minimum permissions needed — nothing more. If a user only needs to view a document, grant read without write. Also called "need-to-know".
Attack Surface Minimisation
Shrink what an attacker can reach. Disable unused services, close unneeded ports, remove dead code — fewer features mean fewer latent vulnerabilities.
Centralized Parameter Validation
Validate every input through one shared library so malformed data can't reach vulnerable code paths. Prevents injection, DoS, and RCE classes at a single chokepoint.
Centralized General Security Services
Consolidate shared security functions — authentication, logging, crypto — into dedicated services. Back them with redundancy so centralisation doesn't become a single point of failure.
Error & Exception Handling
Design to fail safe: a crashed firewall should block, not allow. Never leak sensitive data (stack traces, memory, other users' info) through error messages.

Trust Models

How much should a system trust a user, device, or request that arrives from inside the perimeter? Two dominant philosophies answer this very differently — one assumes insiders are mostly OK, the other assumes nobody is.

🤝
Trust, but Verify
Perimeter-based · Legacy default
The classic model: once a user or device passes the front door (firewall, VPN, corporate network), they're largely trusted. Verification happens at the boundary, and periodic checks validate ongoing behaviour — but internal traffic flows freely.
  • Strong perimeter — firewall, DMZ, VPN
  • Implicit internal trust — LAN is "safe"
  • Periodic audits — catch misuse after the fact
  • Weakness: once an attacker is inside, lateral movement is cheap
🛑
Zero Trust
Never trust · Always verify
Assumes the network is already hostile — inside and out. Every request is authenticated, authorised, and continuously evaluated against identity, device posture, and context, regardless of where it originates. Defined in NIST SP 800-207.
  • Identity-centric — strong auth + MFA on every request
  • Micro-segmentation — least-privilege per resource
  • Continuous evaluation — device health, risk scoring
  • Trade-off: higher operational and engineering complexity
